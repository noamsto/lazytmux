# Remote nested tmux session — design

**Date:** 2026-07-12
**Status:** Design revised after adversarial review, pending spec review → implementation plan
**Branch:** `feat/remote-nested-tmux-session`

## Problem

When you ssh into a remote host that also runs lztmux and start tmux there, you
get a nested tmux: two servers, two prefixes, two stacked status bars, and the
remote session is trapped inside a single local window. There is no way to make
the remote session feel like a first-class local session.

The desired DX: ssh into a host, run `tmux` (or `tmux a`), have lztmux detect
that it is nested-over-ssh, suggest promoting into a dedicated local session,
and — on accept — surface the remote session as a real local session named
`HOST-<session>` that you are switched into.

## Chosen architecture (and why)

Three architectures were considered:

| | Direction | Security | DX | Effort |
|---|---|---|---|---|
| A. Wrapper session (local-initiated) | none | perfect | good; local entry point | low |
| B. Control-mode pull bridge (iTerm2 model) | local pulls | excellent | best (native windows) | high |
| **C. Reverse-socket push (chosen)** | remote pushes | needs mitigation | matches the imagined DX exactly | medium |

Every mainstream seamless-remote tool (iTerm2 `tmux -CC`, WezTerm mux, herdr)
uses local-pull and forwards **no** reverse channel — the secure direction. C is
the road less travelled precisely because remote→local push is the insecure
direction: whoever controls the remote end can drive the local machine.

C was chosen deliberately for the imagined entry point (type `tmux` on the
remote, local session appears), accepting that it is only enabled on hosts where
**root is you**. The security cost is bounded by the restricted-vocabulary
listener below — but see the Security model for the residual risks an
adversarial review surfaced (forced pane-move / forced switch).

### The herdr lesson: restricted control vocabulary, never the raw tmux socket

The raw local tmux control socket, if forwarded, grants a hostile remote-root
arbitrary code execution on the laptop (`tmux new-window 'anything'`). herdr's
"pure socket API" is the model for avoiding this: forward a small
**restricted-vocabulary** control socket served by a local listener, not the
tmux socket. Even a hostile remote-root is then bounded to the listener's verbs
— but those verbs still carry real capability (see Security model), so the
vocabulary and its argument binding must be designed defensively.

## Components

| Component | Location | Responsibility |
|---|---|---|
| **listener** (bash + `socat`) | laptop, systemd user service | `socat UNIX-LISTEN:<sock>,fork` dispatching each connection to a bash handler. Validates the restricted protocol and translates verbs into local tmux commands. The only thing that touches local tmux. Bash + socat matches the repo's bash-first convention and existing systemd-service pattern (`startupSession`, tmux-state timer); the security property comes from the validation logic, not from a compiled binary. |
| `tmux` shim | shell function shipped by the home-manager module, every host | Wraps bare `tmux` / `tmux a[ttach]`. Detects nested-over-ssh, shows the suggestion, speaks the protocol over the forwarded socket on accept. Inert otherwise. |
| ssh_config block | generated by home-manager, `trustedHosts` only | `RemoteForward` the **listener** socket, `SendEnv TMUX_PANE`, `SetEnv LZTMUX_OUTER=1`, `StreamLocalBindUnlink yes`. |
| sshd `AcceptEnv` | companion NixOS module on each remote | Accepts `LZTMUX_OUTER` + `TMUX_PANE` so the forwarded env survives. |
| inner status handling (optional) | tmux.conf | Hides the inner (remote) status bar to avoid double bars. **Session-scoped**, not per-client (tmux `status` has no per-client scope) — see the caveat below. |

## End-to-end flow

1. Local shell is inside tmux pane `%5`. ssh_config for the opted-in host adds
   `RemoteForward <listener-sock>`, `SendEnv TMUX_PANE`, `SetEnv LZTMUX_OUTER=1`,
   `StreamLocalBindUnlink yes`.
2. `ssh HOST` — plain command; ssh_config carries the forward + env
   transparently. `TMUX_PANE` is only present in the forwarded env if you ssh'd
   from inside local tmux (otherwise the local var is unset and `SendEnv`
   forwards nothing).
3. On HOST you run `tmux` (remote `$TMUX` unset). The shim's detection predicate
   (all must hold):
   - `$SSH_CONNECTION` is set (arrived via ssh), and
   - `$TMUX_PANE` is set and matches `%[0-9]+` (you were inside local tmux — this
     is the real nested signal), and
   - a `connect()` + `hello` handshake to the forwarded socket **succeeds**
     within a short timeout (proves the listener is actually alive — file
     existence is not a health signal; see degradation).
4. Detected → prompt `[Y/n]` (`a` = always this host, `x` = never this host;
   choice remembered per host).
5. Accept → shim sends `promote HOST <session> $TMUX_PANE` over the socket.
6. The listener validates, resolves the target client from the pane, then on the
   local tmux: create `HOST-<session>`, move the pane's window into it, switch
   the resolved client to it (see Promote choreography).
7. Shim execs `tmux new -A -s <session>` on HOST.

**Result:** local session `HOST-<session>` holding your live ssh pane (scrollback
preserved), with your client switched into it.

### Detection & round-trip mechanisms

- **`TMUX_PANE` is an opaque round-trip token.** It is meaningless on the remote;
  its only purpose is to travel to the remote and back so the local listener
  learns which local pane initiated this ssh. `SendEnv` only forwards it when it
  exists locally, so its presence doubles as the "was inside local tmux" signal.
- **Re-entrancy guard:** the shim acts only when the *remote* `$TMUX` is unset
  (you're at the remote shell about to start tmux). Once inside the remote tmux,
  `$TMUX` is set, so re-running `tmux` never re-triggers.

### Graceful degradation (corrected)

The shim treats **a completed `hello` handshake** as the only liveness signal —
never file existence, because `RemoteForward` creates the *remote* listening
socket at session setup regardless of whether the *local* listener is running,
and dials the local target only lazily per connection. Therefore:

- Listener down at ssh time → remote socket file may exist, but `connect()`/
  `hello` fails → shim silent, plain nested tmux.
- ssh from outside local tmux → `TMUX_PANE` unset → predicate fails → silent.
- Non-opted host → no forward, no env → silent.
- Version skew → `hello` replies `incompatible` → silent.
- Non-NixOS remote (no `AcceptEnv`) → `LZTMUX_OUTER`/`TMUX_PANE` unset → silent.

### Concurrent ssh to the same host

The remote socket path is per-host (ssh_config tokens offer no per-connection
uniqueness). With `StreamLocalBindUnlink yes`, the most-recent ssh to a host
owns the forward; an earlier shell whose forward was unlinked simply fails the
`hello` handshake and falls back to plain tmux. This is an accepted limitation:
promotion works from the most-recent connection to a given host; older
connections degrade gracefully rather than misbehaving.

## Restricted protocol

Line-oriented over the forwarded unix socket. Two verbs:

- `hello <proto-version>` → listener replies `ok <version>` or `incompatible`.
- `promote <host> <session> <outer_pane>` → the action.

The listener is the trust gate and validates every field before touching tmux:

- **Verb whitelist** — anything not in the set is rejected and logged.
- **Charset/shape** — `host` and `session` match `[A-Za-z0-9._-]{1,64}`;
  `outer_pane` matches `%[0-9]+`. Stops shell/format metacharacters reaching tmux
  commands and bounds session-name flooding.
- **Pane↔connection binding (defense for finding 5)** — before acting, the
  listener verifies `outer_pane` names a real local pane whose current command is
  `ssh` (`tmux display -p -t <outer_pane> '#{pane_current_command}'`), best-effort
  confirming the pane is actually an ssh session rather than an arbitrary pane the
  peer named. If it is not an ssh pane, refuse.
- **Client resolution** — resolve the origin session of `outer_pane`
  (`#{session_name}`), then `tmux list-clients -t <origin-session>`. Require
  **exactly one** attached client; if zero or more than one, refuse with a
  message (multi-attach is ambiguous — the listener cannot know which client
  issued the ssh). This precondition is stated to the user in docs.
- **Idempotency** — if `HOST-<session>` exists, switch the resolved client to it;
  if `outer_pane` is already in it, no-op.
- **Rate limit** — cap promotions per unit time per socket to blunt flooding.

### Promote choreography

Isolate the pane into its own session and follow the client to it:

1. `break-pane` the `outer_pane` into its own window (detached).
2. Create `HOST-<session>` and relocate that window into it
   (`new-session -d` + `move-window`), dropping the placeholder window.
3. `switch-client -c <resolved-client> -t HOST-<session>`.

The exact tmux incantation (`break-pane` cross-session target vs the
`new-session`/`move-window` sequence, and window-index bookkeeping) is the item
to pin against a live tmux server during implementation. Client resolution
(above) is the load-bearing correctness point, not the pane surgery.

## Config surface

```nix
programs.lazytmux.remote = {
  enable = false;                          # master switch (off by default)
  trustedHosts = [ "lab-*" "myserver" ];   # Host patterns that get the ssh_config block
};
```

When enabled, the home-manager module:

- Runs the bash+`socat` listener as a systemd user service (same pattern as
  `startupSession` / the tmux-state timer). Adds `socat` to the closure.
- Generates the ssh_config block for `trustedHosts` (`RemoteForward`, `SendEnv`,
  `SetEnv`, `StreamLocalBindUnlink yes`).
- Ships the `tmux` shim onto PATH (self-disabling when the detection predicate
  fails).

**Cross-layer wrinkle:** forwarding `TMUX_PANE`/`LZTMUX_OUTER` requires the
remote sshd to `AcceptEnv` them — NixOS system config, not home-manager. So the
remote side needs a small companion NixOS module
(`services.openssh.settings.AcceptEnv`, additive to defaults). This fits the
"lztmux on all hosts" plan but means the remote half is not pure home-manager;
non-NixOS remotes degrade to plain tmux (which is the correct, safe fallback).

## Security model (honest)

Bounding measures:

- Restricted-vocabulary listener → no arbitrary command execution.
- Socket `0600` → only remote-you and remote-root can reach it.
- Opt-in `trustedHosts` only; nothing forwarded elsewhere.
- Strict argument validation (verb whitelist + charset + shape).
- Best-effort pane↔connection binding (the pane must be an ssh pane).
- Single-client precondition + rate limiting.

**Residual risk (surfaced by adversarial review, accepted for trusted hosts):**
a peer who can reach the forwarded socket can still, within the vocabulary,
drive UI actions on the laptop — force-move a qualifying local pane into a new
session and force-switch the (single) attached client to it, repeatedly. The
pane-binding and single-client checks reduce but do not eliminate this: it is a
potential UI-hijack / annoyance / mild DoS, **not** arbitrary code execution.
This is why the feature is opt-in per host and intended only for hosts where
root is you. Enabling `trustedHosts` is consent to this residual risk.

## Inner status-bar caveat

Hiding the inner (remote) status bar is **session-scoped**, not per-client: tmux
`status` is a session option with no per-client variant. If you promote a remote
session that a local operator on that box is *also* attached to (same session
name), turning its bar off affects them too. Options: (a) accept session-wide
status-off with this caveat; (b) promote into a dedicated remote session name so
it is never shared. This is a cosmetic nicety, not core to the feature — it may
ship as (a) or be deferred.

## Edge cases

- Listener down → `hello` fails → shim silent.
- ssh from outside tmux → `TMUX_PANE` unset → shim silent.
- Concurrent ssh to one host → most-recent owns the forward; older shells fail
  `hello` → silent.
- Version skew → `hello` `incompatible` → silent.
- Non-NixOS remote → env not accepted → silent.
- Session-name collision (`HOST-<session>` exists) → switch, don't recreate.
- Zero or >1 clients on the origin session → listener refuses (ambiguous).
- `outer_pane` not an ssh pane → listener refuses (binding check).
- Local pane shared a window with siblings → `break-pane` isolates it; siblings
  stay in the original window.
- Detach from the promoted session → moved pane holds the ssh; remote tmux keeps
  running; reattach re-enters `HOST-<session>`.

## Testing

- **bats — shim detection:** env in → act/skip decision, incl. `TMUX_PANE`-unset
  and handshake-fail paths (mirrors `tests/enrich.bats`).
- **bats — listener validation:** reject bad charset, unknown verbs, non-ssh
  pane, zero/multi client; accept the two valid verbs.
- **Integration:** real local tmux + a fake "remote" socket driving `hello` and
  `promote`; assert pane moved, correct client switched, idempotent re-run, and
  refusal on the multi-client case (reuse the existing tmux-shim harness pattern).
- **Manual:** real ssh to a homelab host after `nix build .#default`, including a
  second concurrent ssh to confirm graceful degradation.

## Out of scope (YAGNI)

- Full per-window mirroring (architecture B).
- Local-initiated entry point (architecture A).
- Non-tmux multiplexers.
- Automatic promotion without a prompt.
- Per-client inner status bar (not a tmux capability).
